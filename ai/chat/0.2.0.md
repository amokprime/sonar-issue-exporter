## Question 1: Standard Versioning & Packaging

Your current approach — dump scripts in the repo root, tag releases on GitHub — actually works fine for a tool this small. But there **is** a standard ladder you can climb, and each rung gives you something real. Here's the full picture from simplest to most formal:

### Level 0: What you have now (acceptable, but has gaps)

```
sonar-issue-exporter/
├── export_sonar_issue.py
├── watch_clipboard.py
├── sample.env
├── README.md
└── ai/chat/
```

**Problems with this:**
- Dependencies are only documented in prose in the README (`pip install html2text pyperclip`). The `requests` dependency isn't even mentioned there — it's just imported with no install hint.
- No version number anywhere machine-readable. Your `ai/chat/0.1.2.md` suggests you're at 0.1.2, but nothing in the code knows that.
- "Double-click to run" on Windows works because `.py` files are associated with Python. On Linux/macOS users need `chmod +x` and the shebang (which you have — good).
- `pip install` from GitHub isn't possible, though for two scripts that's arguably fine.

### Level 1: Add `requirements.txt` (minimal, high-value)

```
requests>=2.28
pyperclip>=1.8
html2text>=2020.1.16
```

This is the single highest-ROI step. It gives you:
- One-command install: `pip install -r requirements.txt`
- Deterministic builds if you pin versions with `==`
- GitHub Dependabot can scan it for vulnerabilities
- Anyone (or any CI) can reproduce your environment

### Level 2: Add `pyproject.toml` (the modern standard)

This is where Python packaging has consolidated — `setup.py`, `setup.cfg`, and `pyproject.toml` have merged into just `pyproject.toml` as the single source of truth (PEP 517/518/621). For your project it would look like:

```toml
[project]
name = "sonar-issue-exporter"
version = "0.1.2"
description = "Export SonarQube Cloud issues to AI-readable text files"
readme = "README.md"
requires-python = ">=3.8"
license = "MIT"   # or whatever you choose
dependencies = [
    "requests>=2.28",
    "pyperclip>=1.8",
    "html2text>=2020.1.16",
]

[project.scripts]
sonar-export = "export_sonar_issue:main"    # CLI entry point
sonar-watch = "watch_clipboard:main"         # CLI entry point
```

**What this gives you over Level 1:**
- **`pip install .`** works — installs your scripts as command-line tools (`sonar-export`, `sonar-watch`) available anywhere on PATH
- **`pip install git+https://github.com/you/sonar-issue-exporter.git`** works for remote install
- **Machine-readable version** — `sonar-issue-exporter --version` becomes possible, and tools like `pip show` report it
- **Single source of truth** — no more split between README deps and actual deps
- **GitHub Release automation** — you can use `hatch-vcs` or `setuptools-scm` to derive version from git tags automatically

**What you'd need to change in your code:**
- Both scripts currently run logic at import time (top-level code in `export_sonar_issue.py`). You'd need to wrap that in a `main()` function so it doesn't execute on import. Your `watch_clipboard.py` already has `main()` — good.
- `export_sonar_issue.py` would need a refactor: move the top-level script logic into `def main():` and add `if __name__ == "__main__": main()`.

## Version Pins in `pyproject.toml`

You do **not** need to manually track minimum versions, and you also don't need to hardcode them at all. My example was being conservative — here's the range of options:

### Option A: No version pins at all (simplest)

```toml
dependencies = [
    "requests",
    "pyperclip",
    "html2text",
]
```

`pip` will install the latest compatible version. This works fine for a small tool where you're not worrying about conflicting dependency trees. If a new major version of `requests` (hypothetically `3.0`) breaks your script, you'd pin it then: `"requests<3"` or `"requests>=2.28,<3"`.

### Option B: Lower bounds only (what I showed)

```toml
dependencies = [
    "requests>=2.28",
    "pyperclip>=1.8",
    "html2text>=2020.1.16",
]
```

This says "I know version X works, anything newer is fine until proven otherwise." You set these once based on what you've tested with, and you rarely need to bump them upward unless you start using a feature added in a newer version.

### Option C: Exact pins (for apps, not libraries)

```toml
dependencies = [
    "requests==2.32.3",
    "pyperclip==1.9.0",
    "html2text==2024.2.26",
]
```

This is what you'd use in a `requirements.txt` for a deployed app where reproducibility matters. Overkill for your project.

### What Dependabot actually does

Dependabot can scan **both** `requirements.txt` and `pyproject.toml` and will open PRs to bump versions when new releases come out. But here's the important distinction:

| Scenario | Dependabot behavior |
|---|---|
| `requests>=2.28` | Dependabot will **not** open PRs for this. The specifier already allows the latest version, so there's nothing to update. |
| `requests==2.32.3` | Dependabot **will** open a PR when `2.32.4` or `2.33.0` comes out, bumping the pin. |
| `requests` (no pin) | Dependabot has nothing to act on — there's no version to bump. |

So Dependabot is most useful when you **pin exact versions**. If you use loose pins or no pins, Dependabot has nothing to do because pip already resolves to the latest.

### My recommendation for your project

Use **no pins** or **lower bounds only**. You don't have a complex dependency tree, and none of these three packages have a history of breaking changes in minor/patch releases. If something breaks in the future, pin it then. You're not running a production server — you're double-clicking a script.

You only need to engage with version management manually when:
- A new major version breaks your usage (rare for `requests`/`pyperclip`/`html2text`)
- You start using a feature added in a specific version (then set the lower bound)

For Dependabot, even with no pins it still has **one valuable trick**: it will alert you to **security vulnerabilities** in your dependencies regardless of how they're pinned. That's worth enabling even if you never get a version-bump PR from it.

To enable it, create `.github/dependabot.yml`:

```yaml
version: 2
updates:
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
```

This costs you nothing and gives you vulnerability alerts.

## GitHub Releases: What to Upload

**Don't upload anything.** Let GitHub's auto-generated source archives do the work.

Here's why: when you create a Release on GitHub and attach no files, GitHub automatically provides two downloads:

- **`Source code (zip)`** — a snapshot of the entire repo at that tag
- **`Source code (tar.gz)`** — same thing, tarball format

These always reflect exactly what's at the git tag. They're always in sync with the repo. Users who want your scripts just download the zip and run `pip install -r requirements.txt` (or `pip install .` once you have `pyproject.toml`).

### Why uploading individual files is worse

| Concern | Auto source archive | Manual file uploads |
|---|---|---|
| Stays in sync with the repo | ✅ Always | ❌ You might forget to re-upload after edits |
| Includes `pyproject.toml`, `.env`, README, everything | ✅ Yes | ❌ Easy to miss a file |
| Includes `ai/chat/` history | ✅ Yes (if it's in the repo) | ❌ Would need to manually add |
| Doesn't include `.gitignore`d files | ✅ Correct (you wouldn't want `.env` with real tokens) | ❌ Risk of accidentally uploading your real `.env` |
| Zero effort per release | ✅ Just tag and go | ❌ Every release needs manual packaging |

### The only reason to attach files

You'd attach files to a Release only if you're distributing **built artifacts** — for example, a `.whl` file (the built package that `pip` can install directly without cloning). But for your project, that's unnecessary complexity. Users can already:

```bash
# Install directly from GitHub (no download needed)
pip install git+https://github.com/you/sonar-issue-exporter.git@v0.1.2

# Or download zip, extract, and install locally
pip install .
```

### How to create a release

1. Tag your commit: `git tag v0.1.2 && git push origin v0.1.2`
2. Go to GitHub → Releases → **Draft a new release**
3. Select the tag `v0.1.2`
4. Write release notes (you can auto-generate from commits)
5. Publish — that's it, no file uploads needed

The release page will show the two auto-generated source archives and users get everything.

## `.py` vs `.pyc` — always host the `.py` files

Never commit or distribute `.pyc` files. They're compiled bytecode, and here's why they don't belong in your repo:

| Concern | `.py` (source) | `.pyc` (bytecode) |
|---|---|---|
| Human-readable / editable | Yes | No |
| Works across Python versions | Yes | No — a `.pyc` compiled on 3.12 won't work on 3.11 or 3.13 |
| Works across OS architectures | Yes | Can differ |
| Git diff / code review | Yes | Meaningless binary |
| pip handles compilation automatically | N/A | pip compiles `.py` → `.pyc` at install time anyway |

Add `__pycache__/` and `*.pyc` to your `.gitignore` so they never get committed:

```
__pycache__/
*.pyc
```

Python generates these automatically whenever a script is imported or run. They're a local cache, not a distributable artifact. When someone does `pip install .`, pip compiles the `.py` files into `.pyc` in the user's site-packages directory on their machine — that's the correct place for them.

## Does `pip install git+...` work without GitHub Releases?

**Yes, it works fine.** GitHub Releases and `pip install` from git are completely independent mechanisms.

When someone runs:
```bash
pip install git+https://github.com/amokprime/sonar-issue-exporter.git
```

pip does this:
1. Clones the repo (using git) to a temp directory
2. Finds `pyproject.toml` in the root
3. Installs the package from those source files

It doesn't look at the Releases page at all. It just needs the repo to be accessible via git.

### When you'd need a Release or a tag

| Install method | Needs a Release? | Needs a git tag? |
|---|---|---|
| `pip install git+https://github.com/you/repo.git` | No | No |
| `pip install git+https://github.com/you/repo.git@v0.1.2` | No | Yes — it checks out that tag |
| `pip install git+https://github.com/you/repo.git@main` | No | No — just a branch name |
| `pip install sonar-issue-exporter` from PyPI | N/A (needs PyPI upload) | N/A |

So right now, your users can already install with the first or third command — no Release page needed. If you want people to pin to a specific version (recommended), just push a git tag:

```bash
git tag v0.1.2
git push origin v0.1.2
```

Then they can install that exact version:
```bash
pip install git+https://github.com/amokprime/sonar-issue-exporter.git@v0.1.2
```

The GitHub Releases page is just a nice UI on top of tags. You can create a Release from a tag later if you want release notes, but it has zero effect on pip's ability to install your package.
