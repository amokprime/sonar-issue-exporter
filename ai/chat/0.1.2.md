## Bug: PR-scoped issues invisible to the API

Your failing URLs contain `pullRequest=5`:

```
?pullRequest=5&open=AZ3wXREjCNL_GdnX3Aq_&id=amokprime_linebyline
```

The SonarCloud `issues/search` API **only returns main-branch issues by default**. Issues that exist solely on a pull request (or a non-main branch) are completely invisible unless you explicitly pass the `pullRequest` (or `branch`) parameter. The script was ignoring those URL parameters entirely.

## Fix (2 changes)

### 1. Extract `pullRequest` and `branch` from the URL (lines 83–84)

```python
pull_request  = qs.get("pullRequest", [None])[0]   # e.g. "5"
branch        = qs.get("branch", [None])[0]         # e.g. "feature/foo"
```

The script now also prints them when present:
```
Component : amokprime_linebyline
Issue     : AZ3wXREjCNL_GdnX3Aq_
PR        : 5
```

### 2. Pass them to the API call (lines 119–126)

```python
search_params = {
    "issues": issue_key,
    "componentKeys": component_key,
}
if pull_request:
    search_params["pullRequest"] = pull_request
if branch:
    search_params["branch"] = branch
```

Previously the API call always used just `{"issues": ..., "componentKeys": ...}`, which only searched the main branch. Now it includes the PR/branch scope when the URL provides it.
