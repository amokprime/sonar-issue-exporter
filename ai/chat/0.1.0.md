How to easily export reports from https://sonarcloud.io? The UI is very "busy" with numerous buttons and tabs that each report is split into

---
Navigating the SonarCloud interface for data export can indeed feel overwhelming due to its feature-rich design. The most direct and customizable way to extract data is through the Web API, which allows you to build reports tailored to your specific needs.

Here are the most practical methods to export your reports, organized by technical ability:

| Method | Best For | Easiest Setup | Pros | Cons |
| :--- | :--- | :--- | :--- | :--- |
| **Built-in Enterprise Reports** | Enterprise plan users needing quick, presentation-ready PDFs. | ✅✅✅ | Automatic, professional formatting, includes PDF/CSV. | Limited to Enterprise Plan, less customizable. |
| **Third-Party Plugins (e.g., bitegarden)** | Non-Enterprise users wanting simple one-click PDF/Excel exports. | ✅✅ | Quick setup, no coding, works with any SonarCloud plan. | Tool-specific dependency, may involve additional cost. |
| **SonarCloud Web API** | Anyone needing full control and custom data extraction (JSON). | ✅ | Highly customizable, free, works with any plan. | Requires technical know-how (API calls, scripting). |

### ⚙️ Method 1: Built-in Reporting (Enterprise Plan Only)
If your organization uses the **Enterprise plan**, reporting features are built directly into the UI:
* **Project PDF Reports**: Generate a summary PDF directly from your project's "Project Information" page.
* **Regulatory Reports**: Download a comprehensive `.zip` file containing a summary PDF and detailed CSV files, located via the 'Download' button on your project's "Branch Summary" page. These are especially useful for compliance audits.

### 🔌 Method 2: Third-Party Plugins
For a simpler, UI-driven download option on non-Enterprise plans, the **bitegarden** plugin is a popular choice. It allows you to download ready-to-use **PDF or Excel reports** in seconds with all your project's code quality and security data. You can try it by downloading the self-executing JAR file and following their quick-start guide.

### 🛠️ Method 3: Using the Web API (Most Flexible)
The SonarCloud Web API is a powerful interface that gives you programmatic access to your data. It's free and works with any plan, but it requires you to write API requests.

The basic workflow is:
1.  **Get an Access Token**: Generate a token from your SonarCloud account security settings (`My Account` > `Security`).
2.  **Make API Requests**: Use tools like `curl` to call specific API endpoints. For example, to fetch a project's measures (like bugs, vulnerabilities, etc.), you can use the `api/measures/component` endpoint. The documentation can be found at `https://sonarcloud.io/web_api`.

Here's a practical example to get you started:

```bash
# Example: Get a project's metric values
# Replace $PROJECT_KEY with your project key and $TOKEN with your token.
curl --request GET \
  --url "https://sonarcloud.io/api/measures/component?component=$PROJECT_KEY&metricKeys=bugs,vulnerabilities,code_smells" \
  --header "Authorization: Bearer $TOKEN"
```
The API returns data in **JSON format**, which you can then process into any report format you need.

#### 💡 Pro Tip: Reverse Engineering the API
A highly effective way to learn the API is to "eavesdrop" on the SonarCloud web UI. Open your browser's Developer Tools (usually F12), go to the "Network" tab, and then interact with the SonarCloud interface. You'll see the exact API calls the UI is making. This is the best way to find the correct endpoint for the data you want.

### 📊 Summary
Your choice depends on your specific situation:
* **Enterprise Plan**: Use the built-in PDF or CSV reports for immediate, polished results.
* **Non-Enterprise & Quick**: Explore the **bitegarden** plugin for an easy, UI-based export.
* **Full Control**: Use the **SonarCloud Web API**. It's free and incredibly powerful, but requires the most initial setup.

If you're comfortable with a bit of scripting, the Web API is the most flexible and cost-effective way to get your data exactly how you want it. What specific information are you hoping to include in your reports? Knowing that could help narrow down the best approach.

### ⚙️ 2. Flexible & Programmatic: The Web API (Single or Bulk)

#### Step-by-step for your example issue:

**a) Get the issue details**  
Use the `/api/issues/search` endpoint with your issue key (`AZ3SwtUrqA9SeRODYM9b`).

```bash
curl --request GET \
  --url "https://sonarcloud.io/api/issues/search?issues=AZ3SwtUrqA9SeRODYM9b&componentKeys=amokprime_linebyline" \
  --header "Authorization: Bearer YOUR_TOKEN"
```

The response includes:
- `message` – short issue description
- `rule` – the rule key (e.g., `squid:S106`), which you can use to fetch detailed “Why” and “How”.
- `textRange` – `startLine`, `endLine`, `startOffset`, `endOffset` (the precise location in the source file).
- `component` – the file path.

**b) Get the “Why” and “How”**  
With the rule key from above, call:

```bash
curl --request GET \
  --url "https://sonarcloud.io/api/rules/show?key=squid:S106" \
  --header "Authorization: Bearer YOUR_TOKEN"
```

The `rule.description` field (in HTML) contains the **“Why is this an issue?”** text.  
The `rule.mdNote` or `rule.htmlNote` often provides the **“How can I fix it?”** guidance.

**c) Where is the issue? (the annotated code)**  
The API **does not return the graphically annotated code with the red eyes**. It does give you the exact line range. If you need the visual, the easiest is to keep using a screenshot/PDF for that part.

If you want to *automate* the creation of a report, you can:

1. Fetch the source code of the file via the API (`/api/sources/raw?key=...`).
2. Extract the lines specified by `textRange`.
3. Place them in a code block (maybe with line numbers) in your report.
4. Add the “Why” and “How” text below.

This gives you a textual representation, but the visual annotations are lost. That’s the trade‑off for full automation.

### ✅ Fields an AI actually **needs** to diagnose the issue
- `rule` → tells which rule was violated (e.g., `javascript:S3776`)
- `message` → human-readable description of the problem
- `severity` & `type` → to gauge impact (e.g., CRITICAL, CODE_SMELL)
- `component` (file path) → `docs/index.html`
- `line`, `textRange` → exact location in the file
- `flows` → secondary locations with messages and text ranges; essential for understanding cognitive complexity breakdown
- `impacts` (optional but helpful) → shows software quality affected (MAINTAINABILITY)
- `cleanCodeAttribute` / `cleanCodeAttributeCategory` → nice context but not strictly required

Everything else can be **safely redacted** without impairing the AI’s ability to analyze the code issue.

### 🚫 Fields that should be redacted (and why)
| Field | Reason for redaction |
| :--- | :--- |
| `key` | Internal SonarCloud issue identifier – not needed for code analysis. Keeping it might leak linkage to your specific instance. |
| `hash` | Internal hash; no diagnostic value. |
| `assignee` | Often a user ID or email (e.g., `amokprime-OmKhI@github`). Could be personal data. Not needed. |
| `author` | Git author email (e.g., `blahblahmygithubemail` – you already redacted). Even if it’s in public Git log, sharing it in a report adds no diagnostic value and risks spreading personal info. |
| `organization` (in issues & components) | Your SonarCloud organization name (`amokprime`). While likely public, it’s not needed for the AI to understand the code issue. Redact if you prefer anonymity. |
| `uuid` (in components & issue) | Internal identifiers – no use for diagnosis. |
| `lastChangeAnalysisUuid` | Internal analysis tracking – irrelevant. |
| `creationDate`, `updateDate` | Timestamps – not needed to fix the code. Could be used to infer activity patterns but have no impact on the issue itself. |
| `effort`, `debt`, `debtTotal`, `effortTotal` | Estimated remediation time – not needed for the AI to explain the issue; beneficial only for prioritisation. |
| `tags`, `internalTags` | Metadata (e.g., “brain‑overload”) – not required for code analysis. |
| `project`, `projectName` | Project keys/names – the file path already gives enough context. |
| `status`, `issueStatus` | OPEN – implicit; not needed for explaining the violation. |
| `paging` fields | Pagination info – irrelevant for a single issue. |

Below is a Python script that reads the raw SonarCloud API response (from a file or piped via `curl`) and strips out all non‑essential, potentially sensitive data—leaving only what an AI truly needs to understand and fix the code issue.

### `clean_sonar_issues.py`
```python
#!/usr/bin/env python3
"""
Extract only the diagnostic fields from a SonarCloud issues/search API response.
Reads JSON from stdin, writes cleaned issues array to stdout.
"""

import sys
import json

# Fields to keep for each issue – everything else is dropped.
KEEP_FIELDS = {
    "rule",
    "severity",
    "type",
    "component",
    "line",
    "textRange",
    "message",
    "flows",
    "impacts",
    "cleanCodeAttribute",
    "cleanCodeAttributeCategory",
}

def clean_issue(issue):
    """Return a new dict containing only the KEEP_FIELDS of the issue."""
    return {k: issue[k] for k in KEEP_FIELDS if k in issue}

def main():
    try:
        raw = sys.stdin.read()
        data = json.loads(raw)
    except json.JSONDecodeError as e:
        print(f"Error: invalid JSON input - {e}", file=sys.stderr)
        sys.exit(1)

    issues = data.get("issues", [])
    cleaned = [clean_issue(i) for i in issues]

    # Output a pretty-printed JSON array for readability.
    json.dump(cleaned, sys.stdout, indent=2)
    sys.stdout.write("\n")

if __name__ == "__main__":
    main()
```

### How to use it
1. Save the script as `clean_sonar_issues.py` and make it executable if you like (`chmod +x clean_sonar_issues.py`).
2. Pipe your API response directly into it:
   ```bash
   curl --request GET \
     --url "https://sonarcloud.io/api/issues/search?issues=AZ3SwtUrqA9SeRODYM9b&componentKeys=amokprime_linebyline" \

     | python3 clean_sonar_issues.py > minimal_issue.json
   ```

   Or, if you already have the response in a file:
   ```bash

   ```

```python
#!/usr/bin/env python3
"""
Export all three tabs of a SonarCloud issue from a browser URL.

Reads BEARER_TOKEN and RAW_URL from .env (or environment variables).
Saves:
  - where.json  (the stripped-down issue data)
  - why.md      (the "Why is this an issue?" explanation)
  - how.md      (the "How can I fix it?" guidance)

File collisions are avoided by appending an incrementing number.
"""

import os
import sys
import json
import re
import requests
from pathlib import Path
from urllib.parse import urlparse, parse_qs

# Try to use html2text for better Markdown conversion; fall back to simple stripping.
try:
    import html2text
    _HTML2TEXT = html2text.HTML2Text()
    _HTML2TEXT.body_width = 0
    _HTML2TEXT.ignore_links = False
    def html_to_md(html: str) -> str:
        return _HTML2TEXT.handle(html).strip()
except ImportError:
```

## 🧠 Option 2: Fully automatic clipboard watcher (1 click)

A tiny Python script that runs silently in the background, detects when you copy a new SonarCloud link, and automatically fires off the exporter.

### Clipboard watcher script: `watch_clipboard.py`

```python
#!/usr/bin/env python3
"""
Watch the system clipboard for SonarCloud issue URLs and automatically
export them using export_sonar_issue.py.
Run once and leave it in the background. Press Ctrl+C to stop.
"""
import time
import subprocess
import sys
import re

try:
    import pyperclip
except ImportError:
    print("pyperclip not installed. Run: pip install pyperclip", file=sys.stderr)
    sys.exit(1)

EXPORT_SCRIPT = "export_sonar_issue.py"
# Regex to quickly identify a SonarCloud issue URL
URL_PATTERN = re.compile(
    r"https://sonarcloud\.io/project/issues\?.*?open=[A-Za-z0-9_-]+"
)

def main():
    print("Watching clipboard for SonarCloud issue URLs...")
    print("Copy a link to trigger export. Press Ctrl+C to stop.\n")
    last_url = ""

    while True:
        try:
            current = pyperclip.paste().strip()
            if current != last_url and URL_PATTERN.search(current):
                last_url = current
                print(f"🔄 Exporting: {current[:80]}...")
                subprocess.Popen(
                    [sys.executable, EXPORT_SCRIPT, current],
                    creationflags=subprocess.CREATE_NO_WINDOW if sys.platform == "win32" else 0
                )
            time.sleep(1)  # poll every second
        except KeyboardInterrupt:
            print("\nWatcher stopped.")
            break
        except Exception as e:
            print(f"Error: {e}", file=sys.stderr)
            time.sleep(5)

if __name__ == "__main__":
    main()
```

**Setup:**  
1. Install `pyperclip`: `pip install pyperclip`  
2. Run `python watch_clipboard.py` and leave the terminal open (minimised).  
3. Copy a SonarCloud issue URL → the export runs automatically.

It polls the clipboard every second and launches the export script as a separate process, so you can queue multiple issues quickly.

## ⚖️ Trademark considerations

### SonarSource’s trademark policy (the Do Nots)

According to SonarSource’s published trademark guidelines:

> **Do not use a Sonar Mark in your website name, business name, product name, trademarks, domain registrations, social media handles, brand assets, etc.**
> **Do not use a Sonar Mark in a way that implies partnership, sponsorship, or endorsement.**
> **Do not use a Sonar Mark any more than is necessary to factually describe your software. If the expression or sentence has the same meaning without the Sonar Mark, remove it.**

“Sonar” (and its compound marks like SonarQube, SonarCloud, SonarSource) is a registered trademark. Using it **in your project name** (repo name, package name, title) is exactly what SonarSource’s “Do Not” list prohibits.

### Risk levels for your three candidates

- **ExportSonar**: Moderate risk. Combines a generic verb with the trademark. The guidelines say don’t use a Sonar Mark in a product name, but the “Do” section allows factual description in text. A name like this sits in a grey area—it could be read as a product name (problematic) or descriptive text (permitted). The distinction matters.

- **SonarGet**: Higher risk. “SonarGet” reads like an obvious play on `wget` / `curl`-style fetch tools. It’s the kind of name that a casual user might assume is an official SonarSource utility. Even with a prominent disclaimer, the name itself risks “implying partnership, sponsorship, or endorsement”. This is the diciest of the three if you want to avoid any chance of a trademark dispute.

- **SonarCurl**: Higher risk. Same reasoning as SonarGet—it fuses the trademark with `curl`, suggesting an official API wrapper. The connotation of “curl for Sonar” pushes it further into implied-endorsement territory.

### What the compliant tools do

Look at how the established tools handle naming:

| Tool | Name strategy |
| :--- | :--- |
| `sonar-tools` | “sonar” as a descriptor + “tools” (generic); includes disclaimer: “not supported nor endorsed by SonarSource” |
| `sonar-api-getter` | Functional description (not a brand-like name); no attempt to sound like a product |
| `sonarqube-issues-export-to-excel` | Utterly descriptive, unmistakably third-party |
| `bitegarden Report for SonarCloud` | “for SonarCloud” describes what it works *with* (not what it *is*), plus the company name distinguishes it |

None of these use a catchy portmanteau like `SonarGet` or `SonarCurl`. They lean heavily descriptive and, in the case of `sonar-tools`, pair the name with an explicit disclaimer.

### Recommended naming approach

Given SonarSource’s guidelines, a safer pattern is:

> **[Your identifier] + for + SonarCloud**

Examples that flag to users (and lawyers) that this is a third-party tool and not a SonarSource product:

- `sonar-issue-exporter` (descriptive, follows `sonar-tools` precedent)

```python
try:
    import pyperclip
except ImportError:
    print("pyperclip not installed. Run: pip install pyperclip", file=sys.stderr)
    sys.exit(1)

URL_PATTERN = re.compile(r"https://sonarcloud\.io/project/issues\?.*?open=[A-Za-z0-9_-]+")
last = ""

print("👀 Watching clipboard for SonarCloud issue URLs…")
while True:
    try:
        cur = pyperclip.paste().strip()
        if cur != last and URL_PATTERN.search(cur):
            last = cur
            print(f"Exporting: {cur[:90]}...")
            subprocess.Popen(
                [sys.executable, "export_sonar_issue.py", cur],
                creationflags=subprocess.CREATE_NO_WINDOW if sys.platform == "win32" else 0
            )
        time.sleep(1)
    except KeyboardInterrupt:
        print("\nWatcher stopped.")
        break
```
