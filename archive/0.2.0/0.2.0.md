**Existing conventions:**
- **SARIF** (Static Analysis Results Interchange Format, the industry standard): `ruleId → level → message → locations`
- **ESLint**: `ruleId → severity → line → column → message`
- **SonarCloud's own API**: roughly `rule → severity → component → message → line → ...`
- The common thread: **rule identity → location → description → severity/classification → detailed evidence**

The user's insight about one-liners vs. multi-line arrays is the key design principle. Let me map it out:

| Position | Key | Rationale | Lines |
|----------|-----|-----------|-------|
| 1 | `rule` | Primary identity — which rule fired | 1 |
| 2 | `component` | Which file — context for everything below | 1 |
| 3 | `line` | Quick reference number | 1 |
| 4 | `textRange` | Precise character location (small fixed-shape object) | ~5 |
| 5 | `message` | What the issue says — the most human-relevant one-liner | 1 |
| 6 | `severity` | Triage priority | 1 |
| 7 | `type` | Kind of issue | 1 |
| 8 | `cleanCodeAttribute` | Quality attribute | 1 |
| 9 | `cleanCodeAttributeCategory` | Quality category | 1 |
| 10 | `impacts` | Impact details (array of objects) | ~4-6 per item |
| 11 | `flows` | Code flow evidence (potentially large array) | many |

**Reading flow:** *What rule? Where? What does it say? How bad? Detailed evidence.*

That gives a reviewer 9 one-liner values in the first ~15 lines — a full picture at a glance — before the multi-line arrays start.
